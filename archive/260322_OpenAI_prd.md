1. 문서 개요

제품명(가칭)
Goal Focus

문서 버전
v0.1

문서 목적
기존 보상형 뽀모도로 데모를,
“AI 시대에도 생각 없이 흘러가지 않도록 자신의 목표를 세우고 정해진 시간 동안 실행하는 일반인용 목표 집중 앱”으로 재정의한다.

한 줄 정의
사용자가 목표 사진과 오늘의 할 일을 걸어두고, 방해 없이 세션을 수행하며, 기록·공유·연속 보상·자기계약형 금전 커밋까지 연결되는 목표 중심 집중 앱.

2. 제품 배경

현재 레포는 “키워드 기반 사진/명언 보상형 뽀모도로”에 가깝다.
하지만 사용자가 원하는 제품은 단순 타이머가 아니라 다음이 핵심이다.

내 목표를 먼저 세운다
목표를 달성하기 위한 오늘의 할 일을 정한다
정해진 시간 동안 집중한다
집중이 깨지면 즉시 실패 처리된다
연속 성공 시 더 큰 보상을 받는다
내가 만든 목표/뽀모도로를 남과 공유할 수 있다
몇 번 수행했는지 기록이 남는다
장기적으로는 실제 돈을 걸고 실패 시 차감되는 구조도 가능해야 한다

즉, 제품 중심축은 “타이머”가 아니라 “목표 달성 루프” 여야 한다.

3. 문제 정의

일반 사용자는 다음 문제를 겪는다.

목표는 있지만, 지금 당장 무엇을 해야 하는지 모른다.
집중 앱을 켜도 실제 행동보다 꾸미기와 설정에 시간을 쓴다.
중간 이탈을 스스로 합리화한다.
꾸준함이 끊기면 동기와 기록이 동시에 무너진다.
혼자 하면 약하지만, 공개/기록/보상/손실이 있으면 지속 가능성이 올라간다.

19. 정보 구조와 상태 흐름
19.1 핵심 객체

새 제품의 핵심 객체는 다음 순서입니다.

User → Goal → Task → FocusSession → SessionEvent → GoalStats → CommitmentChallenge

즉, 타이머가 최상위가 아닙니다.
타이머는 FocusSession의 한 속성일 뿐입니다.

19.2 세션 상태 머신

권장 상태 전이는 아래와 같습니다.

draft
  ↓
running
  ├─ success
  ├─ fail(tab_hidden | window_blur | idle | timeout | manual_abort)
  ├─ canceled
  └─ expired

현재 레포의 PomodoroTimer는 하나의 전역 상태를 들고 focus → short_break → focus를 반복하다가 4세션 후 completed로 끝나는 구조입니다. 새 구조에서는 이를 “사용자별 session row + 서버 권위 상태”로 바꿔야 합니다.

19.3 점수 계산 규칙

권장 계산식은 아래와 같습니다.

기본 성공 = 10점
2연속 = Spare +5
3연속 = Strike +10
5연속 이상 = Turkey 상태, 이후 성공마다 기본점수 2배
실패 = 0점, 연속 0
하루 목표 달성 = +30

점수는 focus_sessions.points_earned에 저장하고,
목표/사용자 누적치는 집계로 계산하거나 캐시 테이블에 반영합니다.

22. 프론트엔드 이벤트 명세

집중 이탈은 아래 이벤트 우선순위로 처리합니다.

1. visibilitychange(hidden) -> 즉시 fail
2. window blur -> 즉시 fail
3. heartbeat timeout -> expired 또는 fail
4. manual_abort -> fail
5. 네트워크 손실 장기화 -> expired

권장 클라이언트 로직은 아래와 같습니다.

세션 시작
→ 15초마다 heartbeat
→ visibilitychange/blur 발생 시 interrupt API 호출
→ planned_end_at 도달 시 complete API 호출
→ 서버 응답으로 점수/연속/명언 표시

현재 레포의 click-focus-circle 방식은 “주의 환기”에는 쓸 수 있지만, 서비스 기준 집중 판정으로는 약합니다. 그래서 새 구조에서는 보조 연출로만 남기거나 제거하는 편이 낫습니다.
